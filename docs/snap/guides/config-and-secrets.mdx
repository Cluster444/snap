---
title: "Config And Secrets"
description: "How Application config composition and secret references are intended to work."
---

Application is responsible for config composition. Blocks declare config schemas. The Application resolves global config, nested dependency overrides, and secret references before runtime behavior reads config.

## Global config

Every top-level Block gets its key as a top-level config key.

```ts
export default app("chat", {
  blocks: [
    chat({
      retentionDays: 30,
      providerKey: "#!chat.providerKey"
    })
  ]
});
```

At runtime, resolved config is available through the Application's typed config view.

## Dependency overrides

A depending Block may override dependency config without replacing the whole global config.

```ts
chat({
  push: {
    enabled: true
  }
});
```

The exact dependency config API is still being designed. The goal is for app-level config to stay explicit while allowing Blocks to reuse other Blocks.

## Secret references

Config values may reference secrets through strings beginning with `#!`.

```ts
chat({
  providerKey: "#!chat.providerKey"
});
```

If a config string begins with `#!`, Snap treats it as a secret reference. If the referenced secret does not exist, config resolution fails hard.

Literal config strings that begin with `#!` are not supported for now.

## Validation timing

Snap validates Block config when the Application is loaded by a runtime target. Missing required config, invalid config shape, and unresolved secret references fail startup before the platform accepts requests or connections.
