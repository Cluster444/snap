---
title: "Config And Secrets"
description: "How Application config composition and secret references are intended to work."
---

Application is responsible for config composition. Systems declare config schemas. The Application resolves global config, nested dependency overrides, and secret references before runtime behavior reads config.

## Global config

Every top-level System gets its key as a top-level config key.

```ts
export default snap.create((app) => {
  app.use(EmailSystem, {
    from: "hello@example.com",
    apiUrl: "https://sendmail.com/api/v1",
    apiKey: "#!sendmail.apiKey"
  });
});
```

At runtime, resolved config is available through the Application's typed config view.

## Dependency overrides

A depending System may override dependency config without replacing the whole global config.

```ts
app.use(BillingSystem, {
  email: {
    from: "billing@example.com"
  }
});
```

In this scenario, the billing-scoped email config should inherit `apiUrl` from the global email config unless it is overridden.

## Secret references

Config values may reference secrets through strings beginning with `#!`.

```ts
app.use(EmailSystem, {
  apiKey: "#!sendmail.apiKey"
});
```

If a config string begins with `#!`, Snap treats it as a secret reference. If the referenced secret does not exist, config resolution fails hard.

Literal config strings that begin with `#!` are not supported for now.

## Validation timing

Snap validates System config when the Application is loaded by a runtime kernel. Missing required config, invalid config shape, and unresolved secret references fail startup before the platform accepts requests or connections.
