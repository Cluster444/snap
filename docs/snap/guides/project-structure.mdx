---
title: "Project Structure"
description: "Where Snap Apps, Blocks, and Clients live in a workspace."
---

A Snap workspace separates reusable domain packages from deployed products.

Most projects separate three things:

1. Packages that provide reusable Blocks.
2. Apps that compose Blocks into deployed products.
3. Clients that let humans, agents, or other apps interact with an App.

## Recommended layout

```txt
apps/
  chat/
    app.ts
    web/
      app.tsx
      router.tsx
      pages/
      components/
    agent/
      agent.ts

packages/
  engine/
  core/
  leanbod/
    chat/
    push/
```

`apps/[app]/app.ts` exports the Snap App definition. It chooses which Blocks are installed for that product.

```ts
import { app } from "@snaplab/snap";
import { identity } from "@snaplab/core";
import { chat } from "@leanbod/chat";
import { push } from "@leanbod/push";

export default app("chat", {
  blocks: [identity(), chat(), push()]
});
```

The development runtime can load the selected App and run it:

```bash
snap use chat
snap dev
```

The same App definition should later be deployable through different engine-supported targets.

## Clients

Clients live under `apps/[app]/[client-type]`.

Common Client types include:

1. `web` for a browser app.
2. `cli` for a command-line interface.
3. `tui` for a terminal UI.
4. `ios` or `android` for native clients.
5. `agent`, `mastra`, or another agent framework directory for agent Clients.

Client code imports generated client references and framework adapters. It should not import the server-authored internals of a Block directly.

## Blocks

Blocks live in packages because they are reusable domain models.

```txt
packages/leanbod/chat/index.ts
packages/leanbod/chat/documents.ts
packages/leanbod/chat/commands.ts
```

Small Blocks can define their internals inline. Larger Blocks can split by feature area, primitive type, or runtime concern. Snap should not prescribe one internal file layout; keep files readable and keep domain boundaries clear.

## Deployment

Deployment support is expected to live in the engine, not inside each App directory.

An App defines what should run. A deployment target defines how to run it. The first target is expected to focus on a Bun-based deployment that can package a small server and React Client together, while other targets may deploy the App and Clients separately.
