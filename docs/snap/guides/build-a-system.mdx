This guide shows the intended authoring shape for a reusable support System.

<Warning>
  Snap is currently a design-stage concept. The examples on this page describe the intended public API shape.
</Warning>

## Target outcome

We will build an `EmailSystem` that contributes:

1. Config for provider credentials and defaults.
2. A `sendEmail` Job.
3. A `previewEmail` Command.
4. Server behavior through primitive `run` functions.

## 1. Define config

```ts
const EmailConfig = z.object({
  from: z.string().email(),
  apiUrl: z.string().url(),
  apiKey: z.string()
});
```

## 2. Define primitives

```ts
import { command, job, system } from "snap";

const sendEmail = job("sendEmail", {
  input: z.object({
    to: z.string().email(),
    subject: z.string(),
    body: z.string()
  }),
  async run(input) {
    await sendMail(input);
  }
});

const previewEmail = command("previewEmail", {
  input: z.object({ subject: z.string(), body: z.string() }),
  output: z.object({ html: z.string() }),
  async run(input) {
    return { html: renderEmail(input) };
  }
});
```

## 3. Define the System

```ts
export const EmailSystem = system("email", {
  config: EmailConfig,
  jobs: { sendEmail },
  commands: { previewEmail }
});
```

## 4. Install into an Application

```ts
import { snap } from "snap";
import { EmailSystem } from "./email";

export default snap.create((app) => {
  app.use(EmailSystem, {
    from: "hello@example.com",
    apiUrl: "https://sendmail.com/api/v1",
    apiKey: "#!sendmail.apiKey"
  });
});
```

## Dependencies

A product System can depend on support Systems instead of reimplementing their capabilities.

```ts
export const BillingSystem = system("billing", {
  dependsOn: {
    email: EmailSystem
  },
  commands: { createInvoice }
});
```

Reusable Systems should keep their public surface small: config schema, authored primitives, and any typed helpers needed by depending Systems. Runtime details such as provider clients, server bindings, and direct Transport Operations should stay internal unless the System is explicitly exposing an advanced extension point.
