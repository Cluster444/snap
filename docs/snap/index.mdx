Snap is the codename for the next direction of the engine package: a framework for building portable client/server application definitions that can run embedded in one process, distributed across servers and clients, inside deterministic simulation tests, or on platform kernels such as Cloudflare Workers.

<Warning>
  Snap is currently a design-stage concept. The docs describe the intended architecture, not a stable public API.
</Warning>

## The idea

A Snap app is assembled from Systems and built-in Primitives. Systems contribute domain behavior and reusable support capabilities. Primitives provide common execution and data patterns. A server, worker, native host, test harness, or development runtime loads the app definition and runs it with a platform kernel.

## Architecture at a glance

Snap is organized around four architecture concepts:

1. Application: the portable app definition that installs Systems and resolves startup composition.
2. Systems: composable units that group product behavior, support capabilities, config, dependencies, and authored Primitives.
3. Primitives: engine-owned patterns such as Command, Workflow, Job, and Document.
4. Transport: the lower-level runtime boundary for Sessions, Connections, Operations, dispatch, and protocol adaptation.

Interfaces are the client and UI-facing layer built on generated client references, Primitive clients, and framework adapters.

## Phases

Snap has three major design phases:

1. Core App Model: Transport, Primitives, Systems, Application, config, and generated client references.
2. Clients: CLI, TUI, React, Svelte, Solid, native, and agent interfaces built on the generated client surface.
3. Orchestration: Snap Apps that can build, deploy, coordinate, and communicate with other Snap Apps.

These docs focus on the Core App Model before expanding detailed Client and Orchestration material.
