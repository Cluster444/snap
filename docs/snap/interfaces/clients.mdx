---
title: "Clients"
description: "Framework and platform interfaces over Snap client APIs."
---

Clients are interface code built on Snap. Snap provides framework-neutral client APIs and framework adapters that can be used from React, Solid, Svelte, native apps, CLIs, TUIs, agent clients, or embedded environments.

## Client layers

Interfaces can build on:

1. Generated client references from the app definition.
2. Primitive clients for Commands, Workflows, and Documents.
3. Framework adapters for UI-specific ergonomics.
4. Direct Transport Client APIs only for advanced support or platform behavior.

Client code imports generated client references from the application, not server-authored primitive modules:

```ts
import { EmailSystem, ImportSystem, ProjectSystem } from "my-app/client";
```

In development, this module can be produced by the Snap dev runtime and hot reloaded. In production builds, it can be emitted as generated client code and types.

## React example

Primitive adapters may expose hooks such as:

```tsx
const command = useCommand(EmailSystem.commands.previewEmail);
const workflow = useWorkflow(ImportSystem.workflows.importContacts);
const mutation = useMutation(ProjectSystem.documents.project.mutations.rename);
const query = useQuery(ProjectSystem.documents.project.queries.byId);
```

## Workflow state

Workflow adapters can expose run lifecycle state such as acceptance, completion, failure, and cancellation. They should not be the primary place for domain progress.

```tsx
const importContacts = useWorkflow(ImportSystem.workflows.importContacts);
const run = importContacts.run({ importId });

run.status;
run.result;
run.cancel();
```

Domain progress and status should normally be read from Documents owned by the System:

```tsx
const importRun = useDocument(ContactSystem.documents.contactImport, importId);

importRun.status;
importRun.imported;
importRun.total;
```

Detailed client APIs are a later phase of the Snap docs. The first phase focuses on the server-side architecture and authored primitive model so client adapters have stable concepts to build on.

## Interface goal

Any interface in any environment should be able to build on Snap to the extent it wants to or is able to. Web apps, native apps, CLI tools, TUIs, watches, agent clients, and test harnesses should all use the same underlying concepts.
