---
title: "Application"
description: "The portable App definition that composes Blocks."
---

The Application is the engine view of a Snap App definition. It installs Blocks, validates dependency order, resolves config and secrets, extracts Primitive Definitions from authored Primitives, and prepares runtime collections for the target that runs the App.

The Application is not itself the server. A development server, Cloudflare Worker kernel, Bun kernel, embedded runtime, native host, or deterministic test harness loads the Application and runs it.

## Responsibilities

Application owns startup composition:

1. Install Blocks.
2. Validate dependency order.
3. Resolve config and secret references.
4. Extract Primitive Definitions from authored Primitives.
5. Initialize built-in Primitive runtimes.
6. Expose the runtime shape that a platform kernel runs.

The intended public authoring surface is a single app-definition helper:

```ts
import { app } from "@snaplab/snap";
import { identity } from "@snaplab/core";
import { chat } from "@leanbod/chat";

export default app("chat", {
  blocks: [identity(), chat()]
});
```

Inlining a small Block may be valid when the Block is app-specific and readable:

```ts
export default app("chat", {
  blocks: [system("chat", {
    documents: { chatRoom }
  })]
});
```

## Collected definitions

Application exposes collected Primitive Definitions through typed runtime collections:

```ts
Application.commands;
Application.workflows;
Application.jobs;
Application.documents;
```

Block-local definitions may remain accessible through their owning Block:

```ts
chat.jobs.sendMessage;
```

Global Application collections may use qualified keys:

```ts
Application.jobs["chat.sendMessage"];
```

## Config composition

Every Block can provide a config schema. Every top-level Block gets its key as a top-level config key. Dependencies may also appear as nested config objects under the depending Block.

```ts
export default app("chat", {
  blocks: [
    chat({
      retentionDays: 30
    })
  ]
});
```

The exact config API is still being designed, but config should be validated before the App starts accepting requests or connections.

## Secrets

Config values may reference secrets with strings beginning with `#!`.

```ts
chat({
  apiKey: "#!chat.providerKey"
});
```

If a config string begins with `#!`, Snap treats it as a secret reference. If the referenced secret does not exist, config resolution fails hard. Literal config strings beginning with `#!` are not supported for now.

## Running an Application

The normal development path is for the Snap CLI to load the exported app definition and choose the development platform kernel:

```bash
snap dev chat
```

Production deployment should use the same App definition with a deployment-specific target. The first target is expected to focus on Bun, while other targets such as Cloudflare will need different runtime behavior.
