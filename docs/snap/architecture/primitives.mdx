---
title: "Primitives"
description: "Engine-owned execution and data patterns built on Transport."
---

Primitives are built-in Snap runtime families. They provide common execution and data patterns so Systems do not have to define raw Transport Operations for common behavior.

## Built-in primitives

Snap currently centers on four primary Primitives:

1. Command: short-lived server behavior invoked by a Client.
2. Workflow: longer-running server behavior that coordinates durable domain work.
3. Job: background work outside direct Transport handling.
4. Document: local-first data with schema, mutations, journaling, and replication.

Client-facing Primitives generally assume an identified session. Support Systems that need anonymous or pre-auth behavior should use direct request-style Transport Operations instead.

## Internal runtime concept

`Primitive` is an internal engine concept. Application authors define Commands, Workflows, Jobs, and Documents through public primitive functions. They do not define new Primitive types for now.

```ts
command("previewEmail", { /* ... */ });
workflow("importContacts", { /* ... */ });
job("sendEmail", { /* ... */ });
document("project", { /* ... */ });
```

## Definition anatomy

Each Primitive supplies:

1. A Definition shape with a stable `key`.
2. A public authoring function such as `command(...)` or `document(...)`.
3. A runtime that collects definitions from installed Systems.
4. Optional server behavior, such as `run` for Commands and Workflows.
5. Optional Transport operations.
6. Optional Client APIs and UI adapters.
7. Optional durability behavior.

Authored primitive objects may contain server-only behavior. Client applications should import generated client references, not server-authored primitive modules.

## Authored object shape

The public authoring functions may return richer authored objects than the extracted Definition alone:

```ts
const cancelEmail = command("cancelEmail", {
  input,
  output,
  run(input) {
    return cancel(input);
  }
});

cancelEmail.definition;
cancelEmail.run;
```

The `definition` is the shared shape Snap can collect and project into generated client modules. `run` is server behavior and should not be imported directly by client applications.

## Collection

Systems contribute authored Primitives through hard-coded slots:

```ts
system("email", {
  commands: { previewEmail },
  workflows: { importContacts },
  jobs: { sendEmail },
  documents: { emailTemplate }
});
```

Primitive keys are authored locally inside a System. During Application initialization, the Application extracts Primitive Definitions and may qualify keys with the owning System key for global lookup and wire protocol identity.

```ts
EmailSystem.jobs.sendEmail;
Application.jobs["email.sendEmail"];
```

## Runtime ownership

Primitive runtimes are part of Snap. Systems do not install `CommandSystem`, `WorkflowSystem`, `JobSystem`, or `DocumentSystem`. These runtimes are always available when the Application initializes.
