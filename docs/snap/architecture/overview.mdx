The engine is the layer underneath Snap's user-facing App, Block, and Client model.

Most app builders should start with Apps, Blocks, and Clients. Engine concepts explain how Snap can collect Block definitions, expose client surfaces, and run the same App through different runtime or deployment targets.

## Architecture

1. Application: the portable Snap App definition and startup composition root.
2. Blocks: reusable domain capabilities installed into an App.
3. Primitives: engine-owned execution and data patterns built on Transport.
4. Transport: the platform-agnostic runtime boundary for Sessions, Connections, Operations, dispatch, and protocol adaptation.

## Dependency direction

Higher layers build on lower layers:

```txt
Application
Blocks
Primitives
Transport
```

Transport does not know about Blocks. Blocks usually do not use Transport directly. Primitive runtimes adapt common patterns onto Transport so most application authors work with higher-level concepts.

Clients sit outside this engine architecture. They use the client APIs exposed by the App surface.

## Why this matters

This layering is what lets a Snap App run in different shapes:

1. Embedded in one process for local apps or tests.
2. Distributed across HTTP/WebSocket infrastructure.
3. Simulated deterministically inside a single process.
4. Eventually orchestrated as a fleet of many Snaps.

## App-builder path

Most app builders start at the top of the architecture:

1. Export a Snap app definition.
2. Install reusable Blocks.
3. Define Commands, Workflows, Jobs, and Documents inside those Blocks.
4. Build interfaces against generated client references.
5. Let a selected runtime kernel run the app.

Direct Transport usage is an advanced escape hatch for Blocks and platform-level behavior.
