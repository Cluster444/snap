---
title: "Systems"
description: "Composable domain units that contribute definitions to a Snap App."
---

Systems are the main application and package authoring unit in Snap. A System groups domain behavior, configuration, dependencies, server bindings, and authored Primitives.

Systems usually fall into two categories:

1. Application Systems: product-specific domains such as chat, identity, clients, nutrition, training, billing rules, or project management.
2. Support Systems: reusable capabilities that many applications can install, such as email, push notifications, billing providers, communications, auth, or observability.

## What Systems own

A System can own:

1. Config schema.
2. Dependencies on other Systems.
3. Commands.
4. Workflows.
5. Jobs.
6. Documents.
7. Server bindings.
8. Rare direct Transport Operations for advanced support behavior.

## Composition

Systems cannot provide definitions directly to other Systems. Instead, Systems include other Systems as dependencies.

```ts
export const BillingSystem = system("billing", {
  dependsOn: {
    email: EmailSystem
  },
  commands: {
    createInvoice
  }
});
```

Primitive slots are object-shaped in the primary authoring style. The object property gives ergonomic typed access, and the authored primitive key remains the runtime identity.

```ts
export const EmailSystem = system("email", {
  jobs: {
    sendEmail: job("sendEmail", { /* ... */ })
  }
});
```

If the slot key and primitive key differ, Snap should reject the mismatch unless an explicit aliasing feature exists.

Dependency-scoped access is intended to allow ergonomic references such as:

```ts
BillingSystem.email.jobs.sendEmail;
```

The primitive slot remains visible to avoid collisions between Commands, Jobs, Workflows, and Documents.

## File organization

A System must export a System Definition. Its internal file layout can stay proportional to its complexity.

```txt
packages/chat/chat/index.ts
packages/chat/chat/documents.ts
packages/chat/chat/commands.ts
packages/chat/chat/workflows.ts
packages/chat/chat/jobs.ts
```

Small Systems can define primitives inline. Larger Systems can split by primitive type, feature area, or runtime concern.

## Direct Transport usage

Systems may define Transport Operations directly when Primitives are not enough. This should be rare for product Systems. Frequent direct Transport usage probably means Snap is missing a Primitive or a higher-level pattern.

Direct Operations are most useful for support Systems that need pre-auth or anonymous behavior, such as auth bootstrap, registration, password reset, or health checks. Most client-facing primitives operate through identified sessions.

```ts
const health = request("health", {
  input: z.void(),
  output: z.object({ ok: z.boolean() }),
  run() {
    return { ok: true };
  }
});

export const HealthSystem = system("health", {
  operations: { health }
});
```

## Packaging

Long-term, Systems should be downloadable and composable units, likely through npm packages. A reusable email, billing, notification, or collaboration System should feel like a package that can be snapped into an Application.
